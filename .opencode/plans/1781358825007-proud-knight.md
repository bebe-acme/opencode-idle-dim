# Plan — Cleanup the heat + MCP reorg + kill the screensaver + orchestrator review

## Why (diagnosis, measured on Beib's Mac)
Beib saw cores pegged / load 12-15 / 76-90°C while "no opencode tab was working". Root causes, in order:
1. **3 orphaned `@aaronsb/google-workspace-mcp`** (PPID 1 = their opencode died; ~24h CPU each, ~75-90% now) = ~3 cores. Cause: global config declares **3** google-workspace MCPs, so every session spawns 3, and that MCP doesn't exit when opencode dies → leaks + loops.
2. **Google Chrome Helper** ~100% (1 core) — unrelated.
3. **11 dimmed sessions animating the screensaver** (alien stepMs=110, progress stepMs=130) → each opencode 13-27% CPU + **iTerm2+WindowServer ~115%** redrawing. Pure waste.
4. **Memory pressure**: 64 GB full + **14.7 GB swap (96%)** + ~30 GB compressor → kernel thrashing = the sustained heat. Source = **24 opencode sessions** (~600 MB-1.6 GB each) + Chrome. **MCP servers are cheap at rest (~2 MB shims); research MCPs cost ~nothing.**

Hardware (for reference): **Apple M5 Pro `Mac17,9`, 18-core = 6 super + 12 performance cores, 64 GB, 307 GB/s**. 76-90°C under load is normal/safe; throttles only near ~100°C+.

## Decisions (locked with Beib)
- **Research MCPs (tavily/firecrawl/perplexity): keep in ALL projects** (free at rest).
- **Playwright + chrome: keep global (as today).** (`mcp-chrome-*` procs are a separate Chrome-extension MCP, NOT opencode — out of scope.)
- **Google Workspace, per project:** `ACME→acme`, `BOMBO→bombo`, `BEBEBEBE→all 3`, **researchs→none**, everything-else→none. Remove all 3 from the global default.
- **Screensaver: delete it entirely** (keep only the cheap dim theme + tab color so `/idle` still parks visually at zero ongoing CPU).
- **Indicator (active/idle awareness): just do the cleanup now**, then deliver an **orchestrator review** (clideck-style tools) so Beib can pick one instead of hand-customizing iTerm2.
- **Orphaned MCPs: kill now + add a reaper + scope.**

### Step 1 — Kill the orphaned MCPs now (immediate ~3 cores back)

```bash

# verify gone:

```

### Step 2 — Orphaned-MCP reaper (prevent recurrence)
- New script `~/.config/opencode/bin/reap-orphan-mcp.sh`: kill node/npm procs whose **PPID==1** and whose command matches the MCP set (`_npx`, `google-workspace-mcp`, `@modelcontextprotocol`, `live-mcp-for-obsidian`, `mcp-tavily|firecrawl|perplexity|playwright`). PPID==1 + MCP pattern = orphaned (parent opencode died). Log kills to `~/.local/state/opencode-idle/reaper.log`.
- LaunchAgent `~/Library/LaunchAgents/com.beib.reap-orphan-mcp.plist`, `StartInterval` 600s, runs the script. `launchctl load` it.
- Guardrails: only `kill` (SIGTERM), user-owned procs, never touch PIDs without an MCP-pattern match.

## TRACK 3 — Orchestrator review (deliverable: `docs/orchestrator-review.md`)

Goal: pick a tool that shows **working / idle / needs-you per session at a glance** and works with **OpenCode** (most Claude-only dashboards read `~/.claude` transcripts and won't see opencode). Produce a comparison + recommendation; Beib decides; adoption is a separate plan.

**Candidates (filter by real OpenCode support):**
- **clideck** — local app, multi-agent incl OpenCode, live status via OpenTelemetry (working/idle/waiting), native terminals, mobile relay. *Strong fit.*
- **code-orchestrator / "Argus"** — web dashboard, PTY-spawns any CLI (opencode works), status by ANSI-tail (waiting/running/idle/exited), desktop notifications. *Agent-agnostic, strong.*
- **VibeManager** — tmux + xterm.js web, runs OpenCode/bash, system monitoring, phone access.
- **vibe-kanban** — `npx vibe-kanban`, kanban over many agents, centralizes MCP config.
- **tmux-agent-indicator** — keeps a terminal workflow (tmux); has an **OpenCode plugin** using `session.status`/`session.idle`/`permission.*` → pane/window/status-bar state. *Lightest; closest to current workflow.*
- **Zellij status plugins** (zellaude / zellij-attention / claude-code-zellij-status) — tab-name status icons; mostly Claude-hook driven (would need an opencode-event bridge).
- **agent-conductor (xtalax) / yksanjo conductor / agent-dashboard** — powerful (columns, jump-to-iTerm2 tab) but read Claude `~/.claude` transcripts → **likely NOT opencode**; list as "Claude-only / needs adapter".
- **OpenCode-native / DIY-light** — `cool` (Beib's Go TUI already reads the opencode session DB), opencode server/SDK `session.list`, or a tiny event-plugin → flag file → `opencode-iterm-state status` dashboard. Lowest dependency.

**Comparison columns:** OpenCode support · status detection (events / OTel / PTY-tail / transcript) · runs your existing iTerm2 tabs vs spawns its own · jump-to-session · macOS · resource cost · setup effort · mobile · maturity. End with a **recommendation** tuned to "doesn't want to hand-customize iTerm2; wants at-a-glance active/idle + focus."
